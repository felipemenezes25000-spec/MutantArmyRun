# level-gates
## SUMMARY
O subsistema de fases é 100% data-driven: uma única cena Game lê um LevelConfigSO (seed determinística, trackLength, gateSlots[], obstacles[], boss, bossHpMultiplier, startingUnits) e o LevelManager monta a pista em runtime no BeginRun, com spawn/reciclagem POR DISTÂNCIA (60m à frente / 25m atrás do líder) e pooling por prefab. Gates e obstáculos NÃO usam as âncoras do TrackSegment — nascem em Z absoluto do config (gates em x=0 via GatePairView; obstáculos em lane X sorteada pelo RNG da fase). O boss é uma referência direta no LevelConfigSO; o GateManager usa boss.weaknesses/immunities/element para montar pares autoBalance (rota ótima FORTE vs ARMADILHA plausível). O tema visual vem do WorldConfigSO (trackSegmentPrefabs, skybox, cores de atmosfera, props), aplicado no BeginRun. Acima da fase 100, o EndlessLevelGenerator cria LevelConfigSO em memória (seed=índice, 5 slots autoBalance, boss/mundo da Dimensão Final reciclados). Inimigos de pista NÃO existem em nenhuma forma — o análogo mais próximo é o pipeline de obstáculos (slot → pool → impacto por distância → CrowdManager.ApplyObstacleHit).
## KEY CLASSES
### LevelManager (Gameplay/LevelManager.cs)
API: static Instance; Init(); BeginRun(LevelConfigSO); ResetRun(); BuildResult(bool won):LevelResult; float Progress01. Privados-chave: SpawnNextSegment(), SpawnObstacles(level), CheckObstacleImpacts(leaderZ), RecycleObstaclesBehind(leaderZ), InGateSafetyZone(level,z), DrainAll(). Serializados: _spawnAheadMeters=60, _recycleBehindMeters=25, _gateSafetyMeters=6, _laneHalfWidth=2.2, _obstacleImpactHalfWidth=1.1, _obstacleShakeDegrees=0.6. Assina GameManager.StateEntered (Running→BeginRun) e seta GameManager.ResultBuilder=BuildResult. Update: spawn/recicla por distância, CheckObstacleImpacts, RaiseRunProgress (passo 0.5%), e leaderZ>=trackLength → ChangeState(BossFight).
NOTES: RNG: _rng=new System.Random(level.seed); ordem FIXA de consumo: gates → obstáculos → segmentos. RiskResolver recebe RNG DERIVADO (seed*486187739+1). Bônus de meta lidos 1x via GameManager.RunStartBonusProvider (RunStartBonuses: speedRunMult, bossDamage, critChance, extraStartUnits, obstacleLossFactor). Impacto de obstáculo: crowd.ApplyObstacleHit(x, halfWidth) + VFXManager.PlayGateBurst + JuiceEvents.RaiseObstacleHit + Tween.ShakeCamera.
### GateManager (Gameplay/GateManager.cs)
API: static Instance; Init(); SpawnGates(LevelConfigSO, System.Random); Consume(GateConfigSO gate, GateConfigSO rejected); ReleaseAll(). Serializados: _pairPrefab (GatePairView), _autoBalancePool (GateConfigSO[] — pool de configs p/ slots autoBalance, configurado NA CENA). Privados-chave: PickPairForBoss(boss, depth01, rng), CategorizeForBoss(boss), Classify(gate, boss):GateClass{Neutral,Strong,Trap} (estático PRIVADO), IsStrongMutation(m), StrategicElementOf(g), PickWeighted(pool, depth01, rng, exclude), WeightOf (números maiores mais prováveis com depth01→1).
NOTES: Classify: elemento que explora boss.weaknesses=Strong; resistido/imune/igual ao boss.element=Trap; Multiply<1=Trap, >=2=Strong; AddFlat<0=Trap, >=25=Strong; Risk por EV (>=3 Strong, <1.5 Trap). Consume: AddFlat/Multiply→crowd.ReconcileTo(gate.Apply(count)); ClassConvert→ConvertClass; Element→SetElement; Mutation→ApplyMutation; Risk→RiskResolver.Begin. Depois GameEvents.RaiseGateConsumed(new GateResult(gate, rejected, crowd.Count)).
### GateView (Gameplay/GateView.cs)
API: GateConfigSO Config; Bind(GateConfigSO); EnableCollider(); DisableCollider(). Serializados: _config, _label (TMP), _trigger (Collider), _frameRenderer, _frameTrimRenderers[]. OnTriggerEnter: detecta CrowdAnchor via GetComponentInParent → _pair.OnArmyTouched(this).
NOTES: Rótulo SEMPRE de _config.displayLabel via RenderLabel/OnValidate; cor via MPB (_BaseColor/_Color/_EmissionColor*2 p/ bloom); contra-escala do texto.
### GatePairView (Gameplay/GatePairView.cs)
API: Setup(GateConfigSO l, GateConfigSO r, float trackPos) — posiciona em (0,0,trackPos), reseta _consumed; OnArmyTouched(GateView touched) — one-shot, desabilita ambos colliders, chama GateManager.Consume(touched.Config, other.Config); OtherOf(GateView). Serializados: _left, _right (GateView).
NOTES: 1 prefab = par L/R; pooled (defaultCapacity 8, maxSize 32).
### TrackSegment (Gameplay/TrackSegment.cs)
API: public Transform[] gatePairAnchors; public Transform[] obstacleAnchors; public float length=30; public float EndZ {get;set;}
NOTES: ATENÇÃO: gatePairAnchors/obstacleAnchors são CAMPOS MORTOS — o LevelManager nunca os lê; gates/obstáculos nascem em Z absoluto do LevelConfigSO. Segmento é puramente visual.
### LevelConfigSO (+GateSlot, ObstacleSlot) (Core/SO/LevelConfigSO.cs)
API: int levelIndex; int seed; WorldConfigSO world; float trackLength=220; GateSlot[] gateSlots; ObstacleSlot[] obstacles; BossConfigSO boss; float bossHpMultiplier=1; RewardConfigSO winReward; int startingUnits=1. GateSlot: float trackPosition; float depth01; bool autoBalance; GateConfigSO leftGate, rightGate. ObstacleSlot: float trackPosition; GameObject prefab.
NOTES: SO READ-ONLY em runtime; cena Game única para todas as fases.
### GateConfigSO (Core/SO/GateConfigSO.cs)
API: string gateId; GateType gateType (AddFlat=0, Multiply=1, ClassConvert=2, Element=3, Mutation=4, Risk=5); float value; UnitConfigSO unitToAdd; ElementType element; MutationConfigSO mutation; float riskSuccessChance/riskRewardMult/riskFailPenalty; string displayLabel; Sprite icon; Color portalColor; int Apply(int current) → GateMath.Apply (Domain, testado, piso 1).
NOTES: Semântica de TOTAL-ALVO (não delta).
### WorldConfigSO (Core/SO/WorldConfigSO.cs)
API: int worldIndex; string displayNameKey; LevelConfigSO[] levels; BossConfigSO worldBoss; Material skyboxMaterial; GameObject[] trackSegmentPrefabs; AudioClip musicTrack; RewardConfigSO worldClearReward; Color skyTopColor/skyHorizonColor/fogColor/sunColor/ambientColor; Material trackMaterial; GameObject[] propPrefabs.
NOTES: Atmosfera lida pelo WorldAtmosphereApplier no BeginRun; trackSegmentPrefabs sorteados pelo RNG da fase no LevelManager. 10 mundos existem (W01_CampoInicial..W10_DimensaoFinal).
### EndlessLevelGenerator (Core/SO/EndlessLevelGenerator.cs)
API: const DesignedLevelCount=100; ctor(IReadOnlyList<WorldConfigSO>, IReadOnlyList<BossConfigSO>); LevelConfigSO GetOrCreate(int levelIndex) — null se <=100; cache por índice (HideAndDontSave).
NOTES: Build(): seed=levelIndex, trackLength=260, startingUnits=1, obstacles=VAZIO, winReward=null, 5 GateSlots 100% autoBalance (z 30→trackLength-40), boss=Dimensão Final (bossId m10_dimensional_entity), bossHpMultiplier=(1+0.25*voltas)*(1+0.10*passo). NÃO gera obstáculos nem (futuro) inimigos — precisa ser estendido junto.
## EVENT FLOW
Fluxo de uma fase: GameManager.ChangeState(Running) → evento StateEntered → LevelManager.BeginRun(GameManager.CurrentLevel). BeginRun: DrainAll → cria System.Random(level.seed) → configura RiskResolver com RNG derivado (seed*486187739+1) → lê RunStartBonuses via hook GameManager.RunStartBonusProvider (struct: speedRunMult, bossDamage, critChance, extraStartUnits, obstacleLossFactor) → reseta CrowdAnchor/CombatSystem/CrowdManager (ReconcileTo(startingUnits+extraStartUnits)) → ORDEM FIXA DE RNG: GateManager.SpawnGates(level,_rng) → SpawnObstacles(level) (lane X sorteada) → SpawnNextSegment em loop (sorteia world.trackSegmentPrefabs). Durante a corrida (Update): spawn/recicla segmentos por distância; CheckObstacleImpacts (líder cruza Z do obstáculo → crowd.ApplyObstacleHit → VFXManager.PlayGateBurst + JuiceEvents.RaiseObstacleHit + Tween.ShakeCamera); GameEvents.RaiseRunProgress(p) a cada 0.5% (HUD por evento, sem polling). Gate: collider do GateView detecta CrowdAnchor → GatePairView.OnArmyTouched (one-shot do par, desabilita ambos colliders) → GateManager.Consume(chosen, rejected) → efeito puro no CrowdManager (ReconcileTo/ConvertClass/SetElement/ApplyMutation) ou RiskResolver.Begin → GameEvents.RaiseGateConsumed(GateResult{gate, rejected, crowd.Count}) — payload já carrega o REJEITADO (mede rota ótima vs armadilha; base pronta p/ analytics e combos). Fim: leaderZ >= trackLength → GameManager.ChangeState(BossFight). Resultado: GameManager chama o hook ResultBuilder (LevelManager.BuildResult) → LevelResult(levelIndex, won, survivors, damageDealt, coins=0, xp=0, duration) — coins/xp completados pelo GameManager via RunSnapshot (Gameplay não enxerga Meta). Retry/próxima fase: soft reset na MESMA cena (ResetRun→BeginRun), nunca LoadScene.

## RISKS
1) DETERMINISMO DA SEED É CONTRATO: a ordem de consumo do RNG é fixa (gates → obstáculos → segmentos) e documentada como base do Boss Scout e da repro de QA — inserir QUALQUER sorteio novo no meio dessa cadeia muda a pista das 100 fases desenhadas e do endless; novos sistemas devem usar RNG derivado (padrão já estabelecido: RiskResolver recebe new Random(seed*486187739+1)). 2) SOFT RESET NA MESMA CENA: retry/próxima fase reusam tudo via pools (BeginRun→DrainAll); qualquer sistema novo (inimigos, combos) PRECISA de caminho de drain/reset e flags resetadas no Setup (cf. _consumed do GatePairView) — estado residual = bug clássico de pool. 3) ORDEM DE INIT: Init() dos managers vem do bootstrap da cena; GateManager.Init cria o _pairPool ANTES de LevelManager.BeginRun chamar SpawnGates — um novo EnemyManager precisa entrar nessa sequência IInitializable. 4) GERAÇÃO POR FACTORY/EM MEMÓRIA: EndlessLevelGenerator cria LevelConfigSO via CreateInstance e CACHEIA por índice — campos novos no LevelConfigSO ficam no default (null/vazio) nas fases >100 a menos que Build() seja atualizado; o cache também significa que mutar o asset em runtime persiste entre reentradas. 5) ASSETS YAML EDITADOS À MÃO: rebalancear F1-5 é editar guids/floats em YAML (depth01 é redundante com trackPosition/trackLength e precisa ser mantido coerente — o autoBalance pesa por depth01); bossHpMultiplier 0.255→0.64 está calibrado contra maxHp 400 do GolemStone — trocar o boss da F3 para WoodGiant exige recalibrar. 6) ACOPLAMENTO POR SINGLETON + NULL-SAFETY: tudo conversa por Instance estático com guardas de null (greybox-friendly) — fácil de estender, mas a lógica de classificação Strong/Trap está PRIVADA no GateManager e o _autoBalancePool é serializado NA CENA (mudar o pool altera silenciosamente os pares de F5 e do endless inteiro). 7) FRONTEIRAS DE ASSEMBLY: Gameplay não enxerga Meta/Services (hooks via GameManager: RunStartBonusProvider, ResultBuilder, eventos GameEvents/JuiceEvents) — ComboSystem/analytics devem assinar eventos, nunca chamar Services direto; Core não enxerga Gameplay (por isso EnemyDefinition SO deve morar em Core/SO e o comportamento em Gameplay). 8) ÂNCORAS MORTAS DO TrackSegment: usá-las mudaria posicionamento de absoluto→relativo ao segmento, quebrando a garantia de zona de segurança pós-portal calculada por Z absoluto (InGateSafetyZone).

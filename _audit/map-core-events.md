# core-events
## SUMMARY
O subsistema Core é o esqueleto do jogo: GameBootstrap (composition root da cena Boot, prefab [Services]) inicializa managers persistentes em ordem explícita (Save → RemoteConfig → Analytics → Ads → array do Inspector) e expõe um "blackboard" estático (GameBootstrap.Current) com hooks de Ads/IAP/Save para camadas que não se enxergam (UI↔Services). GameManager orquestra a máquina de estados (pilha GameStateStack no Domain, 8 estados: Boot/MainMenu/BossScout/Running/BossFight/ReviveOffer/Victory/Defeat) e resolve o fim de fase via hooks Func/Action preenchidos pelos bootstraps. GameSceneBootstrap repete o padrão na cena Game (Level → Crowd → Gate → Boss → Combat, em Awake). A comunicação entre camadas usa dois buses estáticos de C# events com payloads em struct: GameEvents (9 eventos de dados/estado) e JuiceEvents (2 eventos cosméticos). Tween (corrotinas unscaled, shake por trauma) e Haptics (no-op em desktop) são os utilitários de juice; RcKeys/AdPlacement centralizam strings de Remote Config e placements de rewarded; GameSettingsSO é o catálogo de fases em Resources com endless procedural acima do catálogo desenhado.
## KEY CLASSES
### GameBootstrap (+ IInitializable) (Core/GameBootstrap.cs)
API: interface IInitializable { void Init(); } | class GameBootstrap : MonoBehaviour — static GameBootstrap Current; [SerializeField] GameManager _gameManager; MonoBehaviour _saveService; MonoBehaviour[] _managersInOrder; const MainSceneName="Main", NetworkTimeoutMs=3000. Blackboard: bool AdsRemoved {get;set;}; string ConsentStatus {get;set;}="unknown"; SaveData Save {get;set;}; Action MarkSaveDirty {get;set;}; Func<bool> RewardedAdReady {get;set;}; Action<string,Action<bool>> ShowRewardedAd {get;set;} (placement, cb true=recompensa); Func<bool> SeasonPassOwned {get;set;}; Action<string,float,string,Action<bool>> PurchaseProduct {get;set;} (productId, priceUsd, sourceScreen, onResult)
NOTES: Awake seta Current; async void Start: DontDestroyOnLoad → _gameManager.Init() → resolve IRemoteConfigProvider/IAnalyticsProvider/IAdsProvider via GetComponentInChildren no prefab → InitService(_saveService) → await remoteConfig.InitAsync(online, 3000ms) → analytics.Init(online, ConsentStatus) → ads.Init() se !AdsRemoved → loop _managersInOrder (ordem do array do Inspector É a ordem de init: IAP→Economy→Upgrade→Unit→Reward→Audio→VFX→UI) → LoadSceneAsync("Main") → ChangeState(MainMenu). Singleton em Awake é BANIDO para managers — Instance só dentro de Init().
### GameManager (+ enum DefeatReason) (Core/GameManager.cs)
API: static GameManager Instance; GameState State; LevelConfigSO CurrentLevel; DefeatReason LastDefeatReason; void SetDefeatReason(DefeatReason). Eventos: event Action<GameState> StateEntered; event Action<GameState> StateExited; event Action<int> LevelStarted. Hooks (set pelos bootstraps): Func<bool,LevelResult> ResultBuilder; Func<(int coins,int xp)> RunSnapshot; Action<bool> RunCommitter; Action<int> LevelRewardGranter; Func<int,int> LevelRewardProvider; Action<LevelResult> LevelEndRecorder; Func<bool> ReviveAlreadyUsed; Action ReviveCrowd; Func<RunStartBonuses> RunStartBonusProvider. Métodos: Init(); StartLevel(LevelConfigSO); RestartLevelFromAnyState(LevelConfigSO); GoToMainMenuFromAnyState(); ChangeState(GameState); PushState(GameState); PopState(); OfferRevive(); ResolveRevive(bool). enum DefeatReason { None=0, ArmyWiped=1, BossWon=2 }
NOTES: ResolveEnd(won) roda 1× por fase (_endSelectionDone): constrói LevelResult via ResultBuilder, preenche runCoins/runXp via RunSnapshot ANTES do commit, calcula coinsAwarded = levelReward+runCoins (só vitória) e xpAwarded, chama RunCommitter(won) → LevelRewardGranter → LevelEndRecorder → GameEvents.RaiseLevelFinished. Transições ilegais logam erro e são IGNORADAS (tabela vive no Domain.GameStateStack). Revive: PushState(ReviveOffer) sobre BossFight; ResolveRevive(true) → ReviveCrowd.
### GameSceneBootstrap (Core/GameSceneBootstrap.cs)

## EVENT FLOW
Dois buses estáticos + eventos de instância no GameManager. (1) FLUXO DE ESTADOS: GameBootstrap.Start → GameManager.Init → ChangeState(MainMenu); UI chama GameManager.StartLevel(level) → LevelStarted(int) (analytics) → ChangeState(BossScout) → Running → BossFight → Victory/Defeat. Assinantes de StateEntered/StateExited (Action<GameState>) agem: LevelManager.BeginRun em Running, BossManager.BeginFight em BossFight, UIManager.ShowBossScout em BossScout. Overlays (ReviveOffer, pausa) entram por PushState SOBRE a corrida — Pop devolve a luta intacta. (2) FLUXO DE DADOS (GameEvents, publicador → assinantes típicos): GateManager publica OnGateConsumed(GateResult); CrowdManager publica OnCrowdChanged(count, supplyUsed), OnUnitDied(UnitDeath), OnSupplyOverflow(SupplyOverflow), OnMutationGained(MutationConfigSO); BossManager publica OnBossPhaseChanged(BossPhase{phase 0/1/2, activeWeakness:ElementType}); LevelManager publica OnRunProgress(float 0..1, só em mudança ≥0,5%); EconomySystem publica OnCurrencyChanged(CurrencyChange); GameManager.ResolveEnd publica OnLevelFinished(LevelResult) — UI/Ads/Analytics reagem. Listeners de DADOS rodam síncronos no mesmo frame do Raise; troca de tela é deferida ao frame seguinte. (3) JUICE (JuiceEvents): JuiceController (Gameplay) publica OnBossHitPulse(Vector3) e ObstacleManager publica OnObstacleHit(Vector3); AudioManager (Services) assina — é a ponte cosmética entre asmdefs que não se referenciam. (4) HOOKS (não-eventos): camadas-irmãs (Meta↔Gameplay, UI↔Services) trafegam por Func/Action no GameManager (ResultBuilder, RunSnapshot, RunCommitter, RunStartBonusProvider...) e no blackboard GameBootstrap.Current (RewardedAdReady, ShowRewardedAd, PurchaseProduct, Save, MarkSaveDirty). Fim de fase: EnterState(Victory|Defeat) → ResolveEnd → ResultBuilder → RunSnapshot → RunCommitter → LevelRewardGranter → LevelEndRecorder → RaiseLevelFinished.
## INTEGRATION
(1) Eventos de BossBehavior (OnFightStart/OnHealthPhaseChanged/OnSpecialAttackWarning/OnSpecialAttackExecute/OnWeaknessHit/OnResistedHit/OnDeath): adicionar em Core/GameEvents.cs seguindo o padrão exato `static event Action<TStruct> OnX + static void RaiseX(TStruct)` com structs novos em Core/EventStructs.cs (ex.: BossAttackWarning{attackId, telegraphSeconds, worldPos}, WeaknessHit{ElementType element, float bonus, Vector3 pos}, BossDeath{bossId, overkill, durationSeconds}). OnBossPhaseChanged JÁ É o OnHealthPhaseChanged — reusar, não duplicar. Publicador: BossManager/BossFight (Gameplay), que já assina GameManager.StateEntered(GameState.BossFight) para BeginFight — OnFightStart nasce aí. (2) Feedback FRAQUEZA!/RESISTIU!/BOA ESCOLHA!: se for puramente cosmético, adicionar em Core/JuiceEvents.cs (padrão OnBossHitPulse); se alimentar ComboSystem/analytics, vai para GameEvents. (3) ComboSystem: novo manager IInitializable registrado em GameSceneBootstrap._managersInOrder (Inspector da cena Game, depois de Combat); assina OnGateConsumed (Perfect Gate — GateResult.gate vs rejected já mede rota ótima), OnUnitDied/OnCrowdChanged (No Loss/Clutch), novos OnWeaknessHit (Weakness Hit), OnBossPhaseChanged (Boss Breaker), boss death (Overkill); publica novo OnComboTriggered(ComboResult). Resultado entra em LevelResult — adicionar campos no struct + ctor de 9 args em EventStructs.cs e preencher em GameManager.ResolveEnd (linhas ~188-215) ou via ResultBuilder do LevelManager. (4) Fail reasons ricos: estender enum DefeatReason no FIM de Core/GameManager.cs com ordinais explícitos (None=0/ArmyWiped=1/BossWon=2 já serializados); Gameplay seta via GameManager.SetDefeatReason; ResultScreen lê LastDefeatReason. (5) Tela de vitória DOBRAR: GameBootstrap.Current.RewardedAdReady() para renderizar o botão + GameBootstrap.Current.ShowRewardedAd(AdPlacement.DoubleReward, cb) — base do x2 é LevelResult.runCoins (já separado de coinsAwarded para isso). Cuidado: não re-disparar ResolveEnd (_endSelectionDone protege, mas o crédito extra do dobro deve ir por EconomySystem direto). (6) Morte cinematográfica: Tween.ShakeCamera(amplitude, seconds) + Time.timeScale baixo (tweens default unscaled sobrevivem) + Haptics.Heavy() — convenção já documentada em Haptics.cs como \"golpe final\". (7) BossCollection (álbum) / variantes raras: manager persistente → registrar em GameBootstrap._managersInOrder (prefab [Services]); persistência via GameBootstrap.Current.Save (SaveData é Domain) + MarkSaveDirty(); assina OnLevelFinished/novo OnBossDeath. (8) Inimigos de pista: EnemyManager (Gameplay) em GameSceneBootstrap._managersInOrder; eventos novos em GameEvents (OnLaneEnemyKilled etc.) — NÃO reusar UnitDeath.typeId (byte = índice SoA do exército do jogador, não de inimigos). (9) Tuning remoto novo (combos, bosses, inimigos): consts em Core/RcKeys.cs + defaults no provider; per-boss já existe via RcKeys.BossHpMult(bossId). (10) TutorialDirector: assina GameManager.StateEntered + LevelStarted(int levelIndex) + GameEvents (OnGateConsumed para F1/F2, OnBossPhaseChanged para F3); fases-tutorial 1-5 são LevelConfigSO no catálogo GameSettingsSO.levels — regenerar via MvpContentFactory, não editar o asset à mão.
## GAPS
JÁ EXISTE da missão: (a) fundação de fases de boss — OnBossPhaseChanged com BossPhase{phase 0/1/2, activeWeakness:ElementType} cobre o OnHealthPhaseChanged pedido, com limiares 0.5/0.25 e fraqueza rotativa; (b) ElementType completo (Fire/Ice/Lightning/Poison/Light/Shadow/Metal/Alien) — suficiente para fraqueza elemental do Gigante de Madeira/fogo; (c) fail reason embrionário — DefeatReason{None, ArmyWiped, BossWon} + SetDefeatReason + exibição na ResultScreen, mas só 2 motivos (a missão pede fail reasons ricos); (d) revive no boss completo (OfferRevive/ResolveRevive/ReviveOffer push-state, 1×/fase); (e) infra de Ads/IAP/RemoteConfig/Analytics 100% preparada com providers null/mock atrás de interfaces (IAdsProvider/IRemoteConfigProvider/IAnalyticsProvider) e blackboard pronto, incluindo AdPlacement.DoubleReward para o DOBRAR da tela de vitória e RcKeys.BossHpMult(bossId) para tuning por boss; (f) juice tools prontos para morte cinematográfica — Tween unscaled-por-default (sobrevive a slow motion), ShakeCamera com trauma acumulável, Haptics.Heavy documentado para \"golpe final\"; (g) LevelResult já separa runCoins (base do x2) de coinsAwarded; (h) endless procedural e catálogo de 100 fases. FALTA no Core: (1) TODOS os outros eventos de BossBehavior — não há OnFightStart, OnSpecialAttackWarning, OnSpecialAttackExecute, OnWeaknessHit, OnResistedHit, OnBossDeath em nenhum bus; (2) zero conceito de inimigo de pista — nenhum evento/struct/enum para horda/tanque/atirador/curador (UnitDeath.typeId é índice SoA do EXÉRCITO do jogador); (3) nenhum evento/struct de ComboSystem (Perfect Gate/Weakness Hit/Boss Breaker/Clutch/No Loss/Overkill) e LevelResult não tem campos de combo; (4) sem eventos de feedback FRAQUEZA!/RESISTIU!/BOA ESCOLHA! (JuiceEvents tem só BossHitPulse e ObstacleHit); (5) sem UnitType enum (se EnemyDefinition SO precisar de tipo nomeado, criar em Domain); (6) DefeatReason com só 3 valores vs fail reasons detalhados pedidos; (7) sem GameState para cinematic de morte de boss (se necessário, exigirá mexer na tabela do GameStateStack no Domain); (8) RcKeys sem chaves de combo/inimigos/ataques especiais; (9) nada de BossCollection/variantes raras no Core (precisará de evento de morte de boss + persistência via Save); (10) sem hooks de TutorialDirector contextual (os eventos existentes de gate/fase servem de gatilho, mas nada dedicado).
## RISKS
1) BUSES ESTÁTICOS SOBREVIVEM A CENAS: toda inscrição em GameEvents/JuiceEvents DEVE ser limpa em OnDisable — listener morto de cena anterior = NullReferenceException ou efeito fantasma. Novas features precisam seguir a regra à risca. 2) ORDEM DE INIT É SERIALIZADA NO INSPECTOR: _managersInOrder (prefab [Services] e cena Game) define a ordem — adicionar manager exige editar prefab/cena, e a posição importa (quem publica hook no blackboard deve rodar antes de quem lê). Singleton em Awake é banido; Instance só dentro de Init(). 3) GERAÇÃO POR FACTORY: GameSettingsSO e o conteúdo (LevelConfigSO/WorldConfigSO/BossConfigSO) são gerados pelo MvpContentFactory — edições manuais nos assets podem ser sobrescritas; mudanças de conteúdo das fases 1-5 tutorial devem ir no factory. 4) TABELA DE TRANSIÇÕES NO DOMAIN: ChangeState ilegal só LOGA ERRO E É IGNORADO (estado fica preso silenciosamente) — qualquer GameState novo (ex.: BossDeathCinematic) exige atualizar GameStateStack.Allowed no Domain; ver os workarounds RestartLevelFromAnyState/GoToMainMenuFromAnyState que zeram a pilha para contornar isso. 5) ORDINAIS DE ENUM SÃO CONTRATO: ChestType/DefeatReason têm ordinais serializados/estáveis — estender SEMPRE no fim com valor explícito. 6) HOOKS NULLABLE DEGRADAM EM SILÊNCIO: ResultBuilder/RunSnapshot/RunStartBonusProvider null não quebram, só zeram dados (RunStartBonuses.None) — esquecer de ligar um hook novo no bootstrap passa despercebido em teste rápido. 7) ResolveEnd É ONE-SHOT (_endSelectionDone) e roda DENTRO de EnterState(Victory/Defeat) — features de pós-vitória (DOBRAR, combos) não podem re-disparar o pipeline de crédito; o dobro deve creditar por caminho próprio na Economy. LevelResult é struct mutado após construção — adicionar campos exige tocar nos 2 ctors e em ResolveEnd. 8) async void Start no GameBootstrap: exceção em provider novo some silenciosamente e pode abortar a cadeia de init antes do LoadScene. 9) GameBootstrap.Current/Save podem ser null no boot cedo (Haptics já se defende) — qualquer leitor novo do blackboard precisa de null-check. 10) Camadas não se enxergam (Core não vê Gameplay/Meta/Services/UI; Meta e Gameplay são irmãs): features novas que cruzam fronteiras DEVEM usar eventos ou hooks via Core — referência direta quebra asmdefs. 11) Raise síncrono: listener pesado de evento novo (ex.: combo + UI + analytics no mesmo OnWeaknessHit) roda no frame do hit — manter dados síncronos e visual deferido.

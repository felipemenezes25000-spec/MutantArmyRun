# domain-pure
## SUMMARY
O Domain (MutantArmyRun/Assets/_Project/Scripts/Domain/, namespace MutantArmy.Domain) é uma camada 100% pura: zero UnityEngine, só System (Math, MathF, SHA256, System.Random sempre injetado). Convenção tripla: matemática stateless vira `public static class` (GateMath, RiskGate, CombatMath, FormationMath, InterstitialPolicy, SeasonPassMath, SaveChecksum); estado pequeno vira `public sealed class` (GameStateStack, SupplyLedger, RunWallet, Countdown, ElementChart); dados de fronteira viram `struct` com campos minúsculos espelhando tipos Unity (Float2.x/y, ArenaWave, OverflowPlan, SeasonReward readonly struct). Os testes são xUnit 2.8.1 em net8.0 divididos em 3 projetos (Domain.Gameplay.Tests, Domain.Flow.Tests, Domain.Persistence.Tests) que NÃO referenciam um assembly — cada .csproj inclui os .cs do Domain um a um via `<Compile Include="..." Link="Domain\X.cs" />` (sem glob), sempre com Enums.cs. Nomes de teste em português no padrão Cenario_Condicao_Resultado, com comentários citando doc 12/CANON. Para ComboMath, FailReasonResolver, BossCollectionMath e BossPhaseMath: criar o .cs no Domain seguindo essas regras e adicionar a linha Compile Include no .csproj do projeto de teste certo — o arquivo de teste em si é pego automaticamente pelo glob padrão do SDK.

## EVENT FLOW
O Domain NÃO publica eventos — é 100% pull-based, chamado pela camada Unity (Gameplay/Meta). O fluxo é sempre: manager Unity coleta estado → chama função/método puro do Domain → recebe um valor ou um PLANO em struct → executa o efeito visual/persistente. Exemplos do contrato: CrowdManager chama GateMath.Apply/RiskGate.Resolve e reconcilia atual→total-alvo; SupplyLedger.EnforceCap devolve OverflowPlan (índices a remover + moedas) e o CrowdManager aplica unidade a unidade chamando Remove(); BossConfigSO mapeia UnitConfigSO→id e delega waves para WavePointer.Consume (consumo por ponteiro, cada evento dispara exatamente 1×); ElementChartSO delega para ElementChart (nunca switch hardcoded); AdsManager passa valores de Remote Config como parâmetros de InterstitialPolicy.ShouldShow; o dono do boss ticka Countdown com dt externo. Structs de fronteira usam campos minúsculos para mapear 1:1 com tipos Unity (Float2↔Vector2, ArenaWave↔ArenaWaveEvent do SO, ElementChart.Entry é [Serializable] para o SO). GameStateStack comunica legalidade por retorno bool (transição ilegal = false sem efeito); RunWallet.BuildCommit devolve tupla (long coins, int xp) e auto-zera. Erro de programação vira exceção (Pop em depth 1); entrada inválida de runtime vira clamp/no-op (Countdown.Set negativo, EarnCoins<=0).
## INTEGRATION
Como o dotnet test enxerga o Domain — três csprojs com include explícito por arquivo (SEM glob, SEM project reference): tests/Domain.Flow.Tests/Domain.Flow.Tests.csproj inclui Enums, GameStateStack, Countdown, RunWallet, InterstitialPolicy; tests/Domain.Gameplay.Tests inclui Enums, GateMath, SupplyLedger, ElementChart, CombatMath, CombatAbilities, FormationMath, RiskGate; tests/Domain.Persistence.Tests inclui Enums, SaveData, SaveChecksum, SaveMigration, EconomyMath, ChestMath, MissionMath, SeasonPassMath. Formato exato da linha: <Compile Include="..\..\MutantArmyRun\Assets\_Project\Scripts\Domain\X.cs" Link="Domain\X.cs" /> (separador backslash do Windows — a CI roda windows-latest por isso, ver .github/workflows/tests.yml). RECEITA para um arquivo novo (ex.: ComboMath): (1) criar MutantArmyRun/Assets/_Project/Scripts/Domain/ComboMath.cs em namespace MutantArmy.Domain, sem using UnityEngine, C# 9, APIs netstandard2.1; (2) adicionar UMA linha <Compile Include=...ComboMath.cs Link="Domain\ComboMath.cs" /> no csproj do projeto de teste tematicamente certo (Flow = fluxo de corrida/combo/fail reason; Gameplay = matemática de combate/boss phase; Persistence = coleção/save/economia); (3) criar tests/<Projeto>/ComboMathTests.cs — é compilado automaticamente pelo glob padrão do Microsoft.NET.Sdk (os testes NÃO precisam de include); (4) rodar dotnet test "tests/Domain.Flow.Tests" --nologo (exatamente como a CI). Se criar um 4º projeto de teste, adicionar o step correspondente em .github/workflows/tests.yml. Pontos de conexão das features da missão no código existente: BossPhaseMath → substituir/estender CombatMath.cs:BossPhase (limiares hardcoded 0.5/0.25; OnHealthPhaseChanged = detectar phase(t-1)!=phase(t) com helper puro novo); FRAQUEZA!/RESISTIU! → classificador sobre ElementChart.cs:GetMultiplier/GetBodyMultiplier (>1/<1/==1); telegraph de especial (OnSpecialAttackWarning/Execute) → Countdown.cs:Set/Tick já cobre o timer; horda do Zumbi Titã → CombatMath.cs:WavePointer.Consume; DOBRAR da tela de vitória → RunWallet.cs:BuildCommit(won, multiplier); fail reason → calcular no momento da transição para Defeat via GameStateStack.cs:ChangeState (o chamador decide, o Domain só resolve a razão); novos enums (ComboKind, FailReason, variantes) → Enums.cs (append-only); recompensas de coleção → seguir o padrão tabela determinística de SeasonPassMath.cs:FreeReward/PremiumReward.
## GAPS
JÁ EXISTE no Domain (utilizável pela missão): GateMath/RiskGate (portais, RNG injetado), SupplyLedger+OverflowPlan (F5 supply/decisão real), FormationMath (espiral O(1)), GameStateStack com Victory/Defeat/ReviveOffer e tabela validada, RunWallet com multiplier x2 do rewarded (tela de vitória DOBRAR já tem a metade Domain pronta), Countdown (telegraphs de boss prontos), SaveChecksum/SaveMigration/SaveData (persistência), SeasonPassMath (template de tabela de recompensas), CombatMath.BossPhase (fases 0/1/2 por limiar fixo), ElementChart (multiplicadores de fraqueza/resistência, ciclo Fogo>Gelo>Raio + BodyType — cobre madeira/zumbi/robô via Organic/Undead/Machine), WavePointer+ArenaWave (eventos de wave 1×), InterstitialPolicy (padrão para políticas com defaults de RC), enums Rarity/ElementType/BodyType. Também existem CombatAbilities, EconomyMath, ChestMath, MissionMath (não lidos integralmente, mas incluídos nos csprojs). FALTA para a missão (zero código hoje): ComboMath — nenhum conceito de combo/score (Perfect Gate, Weakness Hit, Boss Breaker, Clutch, No Loss, Overkill) existe em lugar algum; FailReasonResolver — Defeat é só um GameState, nenhum dado de causa de derrota; BossCollectionMath — nada de álbum/coleção/variantes raras de boss (Rarity existe mas nada o usa para bosses); BossPhaseMath rico — BossPhase atual não tem limiares por boss, não detecta transição (necessário para OnHealthPhaseChanged), não modela fraqueza/núcleo exposto/braço destacável; classificador fraqueza/resistiu (>1/<1) para o feedback de texto; nenhum modelo Domain de inimigo de pista (ArenaWave só tem enemyTypeId int — horda/tanque/atirador/curador não existem como dados puros); nenhum teste para GateMath/RiskGate/SupplyLedger/FormationMath/SaveChecksum/SeasonPassMath FOI LIDO em Domain.Flow.Tests (eles vivem nos outros dois projetos — Flow só testa GameStateStack, Countdown, RunWallet, InterstitialPolicy e sanidade de enums).
## RISKS
1) Include explícito por arquivo nos csprojs: esquecer a linha <Compile Include> faz o teste falhar em compilação (tipo inexistente); pior, um arquivo Domain novo que dependa de outro (ex.: RiskGate→GateMath, tudo→Enums.cs) exige incluir AMBOS no mesmo csproj — dependências cruzadas entre arquivos Domain devem ser mínimas e mapeadas. 2) SanityTests.cs asserta Enum.GetValues(GameState).Length == 8 e Boot==0: qualquer estado novo (ex.: BossIntro para morte cinematográfica) quebra o teste DE PROPÓSITO — é um tripwire; atualizar tabela Allowed + teste juntos. 3) Ordinais de enum são contrato com SOs serializados do Unity (comentário em ChestType): novos valores SÓ no fim; nunca reordenar. 4) LangVersion 9.0 + Nullable disable + APIs restritas a netstandard2.1 (Unity 2022): nada de records C#10+, Convert.ToHexString, SHA256.HashData, etc. — o código roda nos dois mundos (Unity e net8.0 dos testes). 5) Caminhos com backslash nos csprojs: CI pinada em windows-latest por isso; trocar para glob quebraria o fatiamento deliberado por projeto temático. 6) GameStateStack: a tabela Allowed é cópia EXATA do doc 12 §4.1 — 'tudo fora dela é bug'; overlays sem linha na tabela só saem por Pop, e Pop em depth 1 lança exceção (estado base nunca sai). 7) Semântica total-alvo do GateMath: qualquer código novo de portal/combo que aplique multiplicador como delta reintroduz o bug 'x2 que triplica'. 8) CombatMath.BossPhase usa <= nos limiares ('atingir o limiar já muda a fase') — BossPhaseMath novo deve preservar essa semântica de fronteira ou os testes/balanceamento divergem. 9) RunWallet.BuildCommit auto-zera (single-commit): integração da tela de vitória não pode chamar duas vezes esperando o mesmo valor. 10) RNG: nunca UnityEngine.Random no Domain — System.Random injetado por parâmetro (padrão RiskGate) para rolls de variante rara de boss serem determinísticos por seed. 11) SaveChecksum.Salt é imutável pós-launch — features de coleção que mexam no formato do save devem passar por SaveMigration, não por mudança de salt/formato do payload.

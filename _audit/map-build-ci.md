# build-ci
## SUMMARY
Projeto Unity 6000.4.8f1 (Unity 6.4) com URP 17.4.0, uGUI 2.0 (TextMeshPro embutido) e Test Framework 1.5.1 — sem Input System novo (activeInputHandler=0, Input Manager legado), sem Addressables e sem nenhum SDK de Ads/IAP/Analytics/RemoteConfig (consistente com a missão de providers null/mock). A arquitetura de assemblies é em camadas: Domain (puro, noEngineReferences) ← Core ← Meta/Services ← UI ← Gameplay, com Editor e dois assemblies de teste (EditMode refs Domain+Core; PlayMode refs Domain+Core+Gameplay+Meta). Build Settings tem 3 cenas (Boot → Main → Game). Android está "meio configurado": min SDK 25, ARM64-only, fullscreen/portrait-friendly, mas ainda com DefaultCompany, applicationIdentifier vazio, scriptingBackend default (Mono — incompatível com ARM64-only), sem keystore, sem ícones e orientação em AutoRotation. CI roda só os 327 testes do Domain via dotnet test (3 csproj xUnit em tests/), Windows + .NET 8; EditMode/PlayMode rodam localmente por exigirem licença Unity. O padrão de PlayMode é forte: GameLoopRig monta a cena por código espelhando os dois composition roots reais, com wiring de [SerializeField] por reflexão e AutoPilot determinístico.

## EVENT FLOW
Bus estático GameEvents (em MutantArmy.Core) com eventos tipados por struct: GameEvents.OnGateConsumed (Action<GateResult>), GameEvents.OnLevelFinished (Action<LevelResult> — campos won:bool, survivors:int), GameEvents.OnSupplyOverflow (Action<SupplyOverflow> — unitsConverted:int, coinsGranted:long, creditado pelo EconomySystem no MESMO frame). GameManager expõe StateEntered (Action<GameState>) e máquina de estados com PUSH: MainMenu → StartLevel(LevelConfigSO) [BossScout] → ChangeState(Running) → BossFight → (wipe c/ revive disponível) ReviveOffer empilhado SOBRE BossFight → ResolveRevive(bool) → Victory/Defeat. Publicadores: GateManager.Consume(GateConfigSO, ?) publica OnGateConsumed e overflow via CrowdManager (SupplyUsed/SupplyCap/Count/ReconcileTo); GameManager publica OnLevelFinished; EconomySystem assina overflow e vitória (GrantLevelReward + commit do RunWallet em ResolveEnd; derrota descarta moedas da corrida). SaveSystem: RecordLevelEnd dispara SaveAsync (save.json em persistentDataPath, com SaveChecksum + backup). Novos eventos da missão (OnWeaknessHit, OnSpecialAttackWarning, OnComboAchieved...) devem seguir esse padrão: structs imutáveis publicados no GameEvents do Core, para UI e Services assinarem sem referenciar Gameplay.

## RISKS
1) Ordem de init dupla: os composition roots (GameBootstrap e GameSceneBootstrap) e o GameLoopRig.Build replicam a MESMA sequência de Init() à mão — qualquer manager novo (ex.: ComboSystem, EnemySpawner) precisa entrar nos três lugares na posição certa, ou os testes PlayMode quebram de formas não óbvias. 2) Wiring por reflexão: renomear um campo [SerializeField] privado de manager quebra o rig (falha explícita, mas fácil de esquecer) — manter os nomes _chart/_defaultUnit/_pairPrefab/_trigger/_left/_right estáveis ou atualizar SetPrivateField em conjunto. 3) Direção Gameplay→UI nos asmdefs: UI não pode referenciar Gameplay; colocar lógica de feedback no lugar errado cria ciclo de assembly (erro de compilação só ao recarregar o editor). 4) Domain com noEngineReferences:true E compilado fora do Unity pelos csproj de tests/ (que usam <Compile Include> com separador Windows): qualquer using UnityEngine em código novo do Domain quebra a CI, e arquivos novos do Domain podem precisar ser incluídos nos csproj. 5) AssetDatabase por caminho fixo nos testes PlayMode: mover/renomear SOs de Assets/_Project/ScriptableObjects/** quebra testes; conteúdo é gerado por 'MAR Tools/Create MVP Content' — bosses/inimigos novos precisam entrar nesse gerador. 6) Time.timeScale: os testes rodam a 8-10x e a missão pede slow motion na morte de boss — implementar o slow-mo de forma que não conflite com timeScale externo (ex.: fator multiplicativo restaurável, ou unscaled time para timeouts), senão os testes de vitória ficam intermitentes. 7) Mono vs ARM64: AndroidTargetArchitectures=2 (ARM64-only) com scriptingBackend default Mono não builda — setar IL2CPP ao mexer no ProjectSettings.asset (arquivo YAML serializado; editar com cuidado ou via editor script). 8) SaveData: campos novos exigem bump de SaveMigration.CurrentVersion + migração + atualização do BuildFullSave dos testes, senão o round-trip passa mas o load em device regride. 9) LogAssert estrito nos PlayMode: qualquer Debug.LogError novo (mesmo benigno, ex.: provider mock reclamando) derruba TODOS os testes de loop.

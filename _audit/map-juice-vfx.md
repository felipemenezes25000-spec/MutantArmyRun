# juice-vfx
## SUMMARY
O subsistema de juice/apresentação do Mutant Army Run é maduro e disciplinado: JuiceController é um orquestrador 100% cosmético que assina o bus estático GameEvents (nunca muta estado) e dispara shakes, FOV punches, flashes via MaterialPropertyBlock, hápticos e bursts; VFXManager é singleton com pooling por prefab (ObjectPool), orçamento global (≤8 sistemas / ≤500 partículas com drop por prioridade), slow motion canônico (timeScale 0.3 por 0.8s com fixedDeltaTime escalado) e telegraph pulsante do especial do boss; CameraRig segue o centróide com damping exponencial framerate-independente, enquadramento dinâmico √n e um estado de boss-framing empurrado pelo BossManager (SetBossFraming/ClearBossFraming) com blend suave. AutoPilot é um piloto de QA/demo (desligado por padrão) que escolhe portais por valor esperado via Domain.GateMath; DevScreenshotRig é um autoteste visual headless com 3 modos de captura offscreen; RiskResolver dramatiza a zona de perigo e resolve via Domain.RiskGate com RNG injetado; WorldAtmosphereApplier aplica skybox/fog/ambiente por mundo no BeginRun. Boa parte da infra de juice da missão JÁ existe — slow-mo de morte de boss, shake, zoom punch, telegraph, confetti, framing de boss — mas falta tudo que é semântico de combate: não há evento por hit (dano de boss é POLLING de Hp a 10Hz), não há conceito de fraqueza/resistência, não há queda cinematográfica do boss (só slow-mo + confetti), e o canal JuiceEvents tem um único evento (OnBossHitPulse, consumido só pelo AudioManager).
## KEY CLASSES
### JuiceController (Gameplay/JuiceController.cs)
API: MonoBehaviour sem API pública (tudo via assinatura de eventos). Campos serializados: _defeatVolume (Volume p/ dessaturação na derrota), _desaturateSeconds=0.35, _bossPulseInterval=0.3, _bossFlashSeconds=0.18, _cascadeMaxPops=8, _cascadeStagger=0.05, _bigMultiplyValue=3, _bigMultiplyShake=1.2, _strongPortalFovPunch=-4, _mutationShake=0.8. Assina: GameEvents.OnGateConsumed/OnSupplyOverflow/OnMutationGained/OnBossPhaseChanged/OnLevelFinished + GameManager.StateEntered. Internos relevantes: PollBossDamage() (polling de BossRuntime.Hp no Update, rate-limited a 0.3s → micro shake 0.5/0.12s + FlashBossRed via MPB + JuiceEvents.RaiseBossHitPulse + Haptics.Light), PunchCameraFov(delta, seconds), CascadePops(newCount), CacheBossRenderers() (busca defensiva por root name 'Boss*'), HandleBossPhaseChanged → ShakeCamera(2.5, 0.5) + FOV punch -7 + Haptics.Heavy.
NOTES: Regra de arquitetura forte: nunca muta estado de jogo; flashes sempre via MaterialPropertyBlock (SetPropertyBlock(null) restaura), nunca instancia material. Usa Tween estático (ScalePop, ShakeCamera, Float, Stop) e Core.Haptics (Light/Medium/Heavy, no-op fora de mobile).
### VFXManager (Gameplay/VFXManager.cs)
API: static VFXManager Instance; void Init() (IInitializable, chamado pelo GameSceneBootstrap); void SlowMotion(float scale, float seconds) — canônico SlowMotion(0.3f, 0.8f), escala Time.timeScale E Time.fixedDeltaTime, tick unscaled, restaura no OnDestroy; void ShowTelegraph(float area, float seconds) — decal no centróide do crowd, anel pulsa escala ±12% e alpha 0.35–0.85 com urgência crescente; void PlayGateBurst(Vector3 pos, Color tint) [prio 2]; void PlayPopBurst(Vector3 pos) [prio 1]; void PlayConfetti() [prio 3, 2 emissores laterais à câmera]; void PlayCoinFanfare(Vector3 pos) [prio 2]; void PlayCrowdDespawnBurst(Vector3 pos) [prio 1]; bool Play(ParticleSystem prefab, Vector3 pos, int priority[, Color? tint]) — pooled, retorna false no drop silencioso. Serializados: _maxActiveSystems=8, _maxLiveParticles=500, _telegraphDecal, _coinFanfarePrefab, _despawnBurstPrefab, _gateBurstPrefab, _popBurstPrefab, _confettiPrefab, _telegraphRenderer.
NOTES: Orçamento global com vítima de menor prioridade; pools por prefab (Dictionary<ParticleSystem, ObjectPool>); hooks null-safe (sem asset = no-op silencioso). Prefabs de juice preenchidos pelo menu 'MAR Tools/Build Juice'. Prioridades atuais: 1=pop/despawn, 2=gate/coin, 3=confetti.
### CameraRig (Gameplay/CameraRig.cs)
API: static CameraRig Instance; void SetBossFraming(Vector3 focusPoint) — empurrado pelo BossManager.BeginFight (focus = meio entre Centroid e boss); void ClearBossFraming(). Serializados: _damping=4 (k exponencial), _baseOffset=(0,9,-7), _bossOffset=(0,11,-15), _bossBlendSeconds=0.8. LateUpdate: segue Centroid (cache anti-NaN), offset dinâmico cresce com raio √n (radius*0.6 up, -radius*0.8 back), blend boss 0..1 com Exp(-4/T·dt), rotação Slerp entre _runRotation e LookRotation(focus).
NOTES: Dois estados apenas: corrida e boss-framing (binário _bossActive com blend contínuo). NÃO conhece BossManager (dados são EMPURRADOS — sem ciclo). Não há estado de 'morte cinematográfica', dolly, órbita nem foco dinâmico em ponto arbitrário fora do par corrida/boss.
### WorldAtmosphereApplier (Gameplay/WorldAtmosphereApplier.cs)
API: void Init() (IInitializable; assina GameManager.StateEntered, aplica só ao entrar em Running); static void Apply(WorldConfigSO world, Light sun) — fonte única da receita: skybox, fog linear 25–80m, ambiente Trilight derivado de ambientColor, sun.color. Serializado: _sunLight.
NOTES: BossFight herda a atmosfera da corrida (não re-aplica). O mesmo Apply é usado pelo factory de editor — mudanças aqui afetam a geração da cena default.
### AutoPilot (Gameplay/AutoPilot.cs)
API: enum SideStrategy { BestExpectedValue, RandomSeeded }; bool Active { get; set; } (nasce false — inerte em produção); SideStrategy Strategy { get; }; void Configure(SideStrategy strategy, int seed). Serializados: _active=false, _strategy, _randomSeed=12345, _extraForwardSpeed=4, _lateralSpeed=10, _laneHalfWidth=2.2. Só age em GameState.Running: empurra CrowdAnchor.transform.position e mira o lado do GatePairView mais próximo à frente; EV calculado via Domain.GateMath.Apply (AddFlat/Multiply/Risk; demais tipos = count atual).
NOTES: É modo demo/QA, não 'attract mode' de produção. Não interage com boss fight (nada a fazer fora de Running), não simula taps de combate, não conhece inimigos de pista. Decisão 1× por par (cache _decidedPair).
### RiskResolver (Gameplay/RiskResolver.cs)
API: static RiskResolver Instance; void Init() (IInitializable); void Configure(System.Random rng) — seed da fase injetada pelo LevelManager, RNG separado da pista; static void Begin(GateConfigSO gate) — inicia zona de 1.5s (1 por vez). Serializados: _zoneSeconds=1.5, _zoneVisual (opcional). Resolução no Update via Countdown do Domain: caminho Titã (IsTitanSacrifice: unitToAdd Legendary com supplyCost≥18 → crowd.SacrificeForTitan) ou Domain.RiskGate.Resolve(rng, chance, rewardMult, failPenalty, count) → crowd.ReconcileTo(target, unitToAdd).
NOTES: A zona é pura dramatização visual (_zoneVisual liga/desliga); odds exibidas = odds usadas (portais honestos, CANON §3.4). Sem feedback de suspense/resultado além do visual on/off — candidato natural a 'BOA ESCOLHA!'.
### DevScreenshotRig (Gameplay/DevScreenshotRig.cs)
API: Propósito: autoteste VISUAL headless. Sem API pública de instância — auto-bootstrap via [RuntimeInitializeOnLoadMethod] quando há flag de linha de comando: -screenshotRun (joga fases 1→4 com AutoPilot, captura a cada 2s + momentos-chave: bossscout/portal/arena/golpe_final/vitoria/derrota), -showcaseRun (tour fase 1 dos 10 mundos), -menuShowcase (tour das telas de meta via UIManager.Push). -shotsDir= customiza saída (default Build\shots). Captura OFFSCREEN 540×960 via RenderPipeline.StandardRequest → PNG; nunca ScreenCapture. Sem flag = 100% inerte.
NOTES: Assina GameEvents.OnGateConsumed/OnLevelFinished + GameManager.StateEntered/LevelStarted. Captura 'golpe_final' no Victory conta com o slow-mo canônico ainda ativo — mudanças no timing da morte do boss afetam esses shots.
## EVENT FLOW
Publicadores → bus: o gameplay publica no bus estático GameEvents (Core/GameEvents.cs): OnGateConsumed(GateResult), OnCrowdChanged(int,int), OnSupplyOverflow(SupplyOverflow), OnMutationGained(MutationConfigSO), OnUnitDied(UnitDeath — ponto de extensão declarado p/ VFX/analytics), OnBossPhaseChanged(BossPhase), OnLevelFinished(LevelResult com .won e .levelIndex), OnCurrencyChanged(CurrencyChange), OnRunProgress(float). Existe um segundo bus dedicado a juice, Core/JuiceEvents.cs, hoje com UM único evento: OnBossHitPulse(Vector3) — publicado pelo JuiceController.PollBossDamage e consumido apenas pelo AudioManager (Services/AudioManager.cs:98). Assinantes de juice: JuiceController assina GameEvents + GameManager.StateEntered (soft reset) e traduz eventos em Tween.ShakeCamera/ScalePop/Float, PunchCameraFov, MPB flashes, Core.Haptics e chamadas ao VFXManager.Instance (PlayGateBurst/PlayPopBurst/PlayConfetti). Caminhos NÃO-evento importantes: (1) dano no boss é detectado por POLLING de BossManager.Instance.Current.Hp no Update do JuiceController (combate agregado roda a 10Hz, não há 1 evento por hit — decisão documentada no header); (2) BossManager chama diretamente VFXManager.Instance.SlowMotion(0.3f, 0.8f) no Die (BossManager.cs:236), VFXManager.Instance.ShowTelegraph(area, windup) no especial (BossManager.cs:184) e CameraRig.Instance.SetBossFraming/ClearBossFraming (BossManager.cs:312/337) — push direto via singleton, sem evento; (3) WorldAtmosphereApplier assina GameManager.StateEntered(Running) e escreve RenderSettings. Regra de ciclo: CameraRig nunca referencia BossManager; JuiceController nunca chama Services; tudo cosmético é unidirecional (gameplay → bus/singleton → apresentação).
## INTEGRATION
1) JuiceEvents (Core/JuiceEvents.cs) é O lugar para os novos sinais semânticos da missão: adicionar OnWeaknessHit/OnResistedHit/OnGoodChoice/OnComboAchieved ao lado de OnBossHitPulse (mesmo padrão Action + Raise*). O BossBehavior modular (OnWeaknessHit/OnResistedHit) deve PUBLICAR aqui; JuiceController, FeedbackTextController e AudioManager assinam. 2) JuiceController.PollBossDamage (JuiceController.cs:301-328) — hoje só distingue 'HP caiu'; é onde o flash vermelho genérico deve se bifurcar em flash de fraqueza (cor do elemento) vs resistência (cinza), mas o ideal é substituir o polling por eventos do novo BossBehavior quando ele existir (o polling foi escolhido porque NÃO havia evento por hit — a missão muda essa premissa). 3) Morte cinematográfica: BossManager.Die (BossManager.cs:236) já chama VFXManager.SlowMotion(0.3, 0.8) — a sequência estendida (shake forte + chuva de moedas + queda) deve ser orquestrada aí ou num novo BossDeathDirector que assine o futuro BossBehavior.OnDeath; VFXManager.Play(prefab, pos, priority) já suporta novos prefabs de VFX (adicionar campos serializados + preencher via 'MAR Tools/Build Juice'). 4) Câmera de morte/foco no núcleo exposto: CameraRig.SetBossFraming(focusPoint) (CameraRig.cs:63) é o gancho existente — para zoom-in dramático na morte ou foco no núcleo do Robô Escorpião, estender com algo como SetFocusOverride(point, offset, blendSeconds) seguindo o mesmo padrão push + blend exponencial. 5) Telegraph dos especiais dos 3 bosses: VFXManager.ShowTelegraph(area, seconds) (VFXManager.cs:84) já cobre OnSpecialAttackWarning — só falta variar forma/cor por boss (hoje é 1 decal genérico no centróide). 6) Feedback 'BOA ESCOLHA!': RiskResolver.Update (RiskResolver.cs:53-81) resolve o veredito mas não publica nada — publicar um evento de resultado (sucesso/falha) ali para o FeedbackTextController. 7) HandleLevelFinished (JuiceController.cs:265) é onde a celebração de vitória deve crescer (combos na tela de vitória são camada UI, mas o espetáculo de mundo entra aqui). 8) AutoPilot (AutoPilot.cs) precisará aprender inimigos de pista: hoje só decide lado por GateConfigSO — novos obstáculos/inimigos exigem estender DecideTargetX/ExpectedValue, senão o DevScreenshotRig (que depende do AutoPilot vencer fases 1→4) quebra.
## GAPS
JÁ EXISTE da missão: slow motion na morte do boss (canônico 0.3×/0.8s com física escalada e restauração à prova de troca de cena — VFXManager.SlowMotion, chamado por BossManager.Die); screen shake parametrizado (Tween.ShakeCamera em 4 intensidades: 0.5 hit, 0.8 mutação, 1.2 portal grande, 2.5 fase de boss); zoom/FOV punch (PunchCameraFov com base FOV estável); flash de dano no boss (MPB vermelho+emissivo); telegraph pulsante do especial do boss com urgência crescente (cobre OnSpecialAttackWarning visualmente); enquadramento cinematográfico do boss (CameraRig.SetBossFraming com blend suave); confetti de vitória com prioridade máxima; dessaturação na derrota; pooling de VFX com orçamento global e drop por prioridade (pronto para receber novos prefabs); hápticos por camada (Light/Medium/Heavy, no-op em desktop); AutoPilot determinístico/seeded (base sólida p/ testes PlayMode dos novos sistemas); pipeline de screenshots headless (3 modos) que já captura arena/golpe final; atmosfera por mundo data-driven; zona de risco com resolução honesta via Domain. FALTA da missão (neste subsistema): reação visual a FRAQUEZA/RESISTIU — não existe o conceito; o flash de hit é sempre o mesmo vermelho, sem variação elemental, e o polling de Hp não consegue distinguir tipo de hit (precisa de evento do futuro BossBehavior); queda cinematográfica do boss — a 'morte' hoje é só slow-mo + confetti + shake de vitória, sem sequência dirigida (câmera não tem estado de morte, sem chuva de moedas no mundo, sem animação de colapso); JuiceEvents tem 1 único evento — o canal existe mas está quase vazio para os 7 hooks do BossBehavior; telegraph é único/genérico (1 decal circular no centróide) — 3 bosses memoráveis pedem formas distintas (linha de laser do Escorpião, área de fogo do Gigante); CameraRig não tem foco em ponto arbitrário (núcleo exposto do Robô Escorpião) nem estados além de corrida/boss; nenhum hook de juice para inimigos de pista (OnUnitDied existe no bus mas o JuiceController NÃO o assina; PlayCrowdDespawnBurst existe mas é por lote); RiskResolver não publica resultado (sem 'BOA ESCOLHA!'); AutoPilot não conhece inimigos de pista nem combate ativo; nenhum hook de ComboSystem (Perfect Gate/Overkill etc. precisariam de novos eventos + assinaturas aqui).
## RISKS
1) Polling vs eventos: JuiceController.PollBossDamage compara Hp por frame com rate-limit próprio; se o novo BossBehavior introduzir eventos por hit E o polling continuar vivo, haverá feedback duplicado (flash+shake 2×) — ao migrar, remover/condicionar o polling, e lembrar que o AudioManager depende de JuiceEvents.OnBossHitPulse publicado por esse polling. 2) CacheBossRenderers acha o boss por NOME de root ('Boss*') — novos prefabs de boss (Zumbi Titã, Robô Escorpião) PRECISAM seguir essa convenção de nome ou o flash/posição do hit silenciosamente param de funcionar. 3) Time.timeScale é global e o VFXManager assume ser o ÚNICO dono do slow-mo (restaura no OnDestroy e via _slowMo unscaled): uma sequência de morte cinematográfica com múltiplas fases de slow-mo deve passar PELO VFXManager.SlowMotion, nunca setar timeScale por fora, ou a restauração quebra (e Time.fixedDeltaTime fica dessincronizado). 4) Ordem de init: VFXManager/RiskResolver/WorldAtmosphereApplier são IInitializable chamados pelo GameSceneBootstrap; Instance só existe pós-Init — novos sistemas que toquem VFXManager no Awake/OnEnable verão null. JuiceController tolera GameManager tardio re-tentando no Update (padrão a copiar). 5) Geração por factory: prefabs de juice são preenchidos pelo menu 'MAR Tools/Build Juice' e a cena default é gravada por factories de editor (WorldVisualFactory usa o MESMO WorldAtmosphereApplier.Apply) — novos campos serializados em VFXManager exigem atualizar o factory, senão ficam null (no-op silencioso, fácil de não notar). 6) Orçamento de VFX: prioridades atuais são 1–3; uma morte cinematográfica com muitos sistemas pode estourar ≤8 sistemas/≤500 partículas e derrubar VFX de prioridade baixa — definir prioridades dos novos efeitos conscientemente (morte de boss deve ser ≥3). 7) DevScreenshotRig + AutoPilot são o autoteste visual: mudanças nas fases 1–5 (tutorial jogável) ou inimigos que o AutoPilot não sabe evitar fazem o -screenshotRun parar de vencer fases e o watchdog de 180s mascara a regressão. 8) Bus estático sobrevive a cenas: todo novo assinante de GameEvents/JuiceEvents DEVE parear += com -= (padrão -= antes de += nos Init repetidos), senão handlers fantasmas pós soft-reset. 9) CameraRig.Instance é setado no Awake sem guard de duplicata; o blend assume exatamente 2 enquadramentos — adicionar um 3º estado (morte) exige refatorar o blend binário _bossBlend com cuidado para não quebrar a saída suave existente.

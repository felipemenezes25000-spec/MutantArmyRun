# ui-meta
## SUMMARY
O subsistema de UI vive em Assets/_Project/Scripts/UI e é organizado em duas cenas: Main (MainMenuController + telas de meta: Tropas/Upgrades/Loja/Mapa/Diário/Settings/Eventos/Passe) e Game (GameUIController + HUD + ResultScreen + PauseOverlay + overlays de celebração). A navegação usa o UIManager com DUAS pilhas (telas com history via Push/Pop; overlays via ShowOverlay/PopOverlay sobre a tela atual), inicializado pelo GameBootstrap (Init explícito, nunca auto-registro). As telas são PASSIVAS: a ResultScreen é preenchida por Bind() pelo GameUIController (glue que assina GameEvents.OnLevelFinished e mostra a tela 1 frame depois); as telas de meta leem tudo pelo MetaBridge (fachada estática que traduz contratos de UI para os singletons do Meta) e disparam eventos (BackRequested, WorldSelected, DoubleRequested...) que camadas externas orquestram. A hierarquia de UI é GERADA por factories de Editor (ProjectSetup, MetaScreensFactory, SystemScreensFactory, RewardScreensFactory) que criam os GameObjects e ligam as refs serializadas — qualquer campo novo [SerializeField] exige atualizar a factory e regerar a cena. Ads/IAP entram pela "blackboard" GameBootstrap.Current (hooks RewardedAdReady/ShowRewardedAd/PurchaseProduct/SeasonPassOwned) — a UI nunca referencia Services diretamente.
## KEY CLASSES
### ResultScreen (UI/ResultScreen.cs)
API: class ResultScreen : UIScreen. Eventos: event Action NextRequested; event Action HomeRequested; event Action<long> DoubleRequested (sobe o doubleBase). Métodos: void Bind(bool won, long coinsDelta, int xpDelta, int survivors, long damageDealt, bool perfect, long doubleBase = -1L, string defeatReason = null); void SetDoubleAvailable(bool available); void ConfirmDoubled(); void CancelDoubleRequest(). Campos serializados: _headerBg (Image verde/vermelho), _titleText, _reasonText (motivo só na derrota), _coinsDeltaText ('+100', DELTA nunca total), _xpDeltaText, _survivorsText, _damageText, _perfectBadge (GameObject), _doubleButton + _doubleButtonLabel ('DOBRAR x2 +N' → 'DOBRADO!'), _nextButton, _homeButton.
NOTES: 100% PASSIVA: não calcula nem credita nada. DOBRAR x2 só dispara DoubleRequested; crédito acontece fora (GrantRunDouble) e volta por ConfirmDoubled/CancelDoubleRequest. Botão dobrar só aparece em vitória com doubleBase>0 E SetDoubleAvailable(true) (rewarded com fill). NÃO há: combos, silhueta do próximo boss, fail reasons ricos.
### GameUIController (UI/GameUIController.cs)
API: MonoBehaviour (glue da cena Game). Serializados: _resultScreen, _feedback (FeedbackTextController), _pauseButton, _pauseOverlay, _settingsScreen. Assina: GameEvents.OnLevelFinished/OnUnitDied, GameManager.StateEntered/LevelStarted, eventos da ResultScreen e do PauseOverlay. Privados-chave: HandleLevelFinished(LevelResult) → ShowResultNextFrame() (coroutine, 1 frame de defer, monta coinsDelta=r.coinsAwarded, doubleBase=r.runCoins, perfect=_unitsLostThisRun==0, defeatReason via DefeatReasonText(), chama Bind + SetDoubleAvailable + UIManager.Push); HandleNextRequested() (GameSettingsSO.NextLevelAfter(current, won) → gm.StartLevel, soft reset na MESMA cena); HandleHomeRequested() (ChangeState(MainMenu) + LoadSceneAsync('Main')); HandleDoubleRequested(long) (GameBootstrap.Current.ShowRewardedAd(AdPlacement.DoubleReward, granted => EconomySystem.GrantRunDouble + ConfirmDoubled / CancelDoubleRequest)); DefeatReasonText() (GameManager.LastDefeatReason: ArmyWiped→'Exército eliminado', BossWon→'O boss venceu', fallback). Também controla a pausa (Time.timeScale 0/1) e dispara PERFECT via _feedback.NotifyArenaReached() ao entrar em BossFight.
NOTES: É AQUI que vitória/derrota viram tela. Qualquer dado novo na ResultScreen (combos, próximo boss) entra por ShowResultNextFrame; qualquer fail reason novo entra por DefeatReasonText.

## EVENT FLOW
Três camadas de eventos: (1) BUS ESTÁTICO GameEvents (Core/GameEvents.cs) — OnGateConsumed(GateResult), OnCrowdChanged(int,int), OnSupplyOverflow(SupplyOverflow), OnMutationGained(MutationConfigSO), OnUnitDied(UnitDeath), OnBossPhaseChanged(BossPhase{phase, activeWeakness}), OnLevelFinished(LevelResult), OnCurrencyChanged(CurrencyChange), OnRunProgress(float). Structs em Core/EventStructs.cs; LevelResult = {levelIndex, won, survivors, damageDealt, runCoins (base do DOBRAR), runXp, durationSeconds, coinsAwarded (TOTAL exibido), xpAwarded}. Regra do projeto: listeners de DADOS (commit/save) rodam antes da UI — o GameUIController defere a ResultScreen 1 frame (ShowResultNextFrame). Assinatura sempre += em OnEnable / -= em OnDisable. (2) EVENTOS DE INSTÂNCIA dos singletons do Meta — UnitManager.OnTroopChanged(string unitId), EconomySystem.OnPlayerLevelUp(int), RewardSystem.OnChestOpened(ChestResult), GameManager.StateEntered/StateExited(GameState) e LevelStarted(int) — consumidos por CelebrationOverlay, ChestRevealOverlay, UIManager e GameUIController. (3) EVENTOS C# DAS TELAS (UI→orquestrador): BackRequested/WorldSelected/SeasonPassRequested/NextRequested/HomeRequested/DoubleRequested/Resume-Restart-Settings-MenuRequested — MainMenuController e GameUIController são os únicos que orquestram (Push/Pop/StartLevel/LoadScene). Serviços entram SÓ pela blackboard GameBootstrap.Current: RewardedAdReady(), ShowRewardedAd(AdPlacement, Action<bool>), PurchaseProduct(id, price, source, Action<bool>), SeasonPassOwned() — providers null/mock já prontos (AdsManager publica os hooks). Fluxo da vitória hoje: ResolveEnd (Core) comita RunWallet → RaiseLevelFinished → GameUIController.HandleLevelFinished → 1 frame → ResultScreen.Bind(won, coinsAwarded, xpAwarded, survivors, damageDealt, perfect=_unitsLostThisRun==0, doubleBase=runCoins, defeatReason) → SetDoubleAvailable(RewardedAdReady) → UIManager.Push.

## RISKS
(1) GERAÇÃO POR FACTORY: toda a hierarquia de UI é construída por scripts de Editor (ProjectSetup.cs:339 BuildResultScreen, MetaScreensFactory, SystemScreensFactory, RewardScreensFactory) que criam GameObjects e atribuem as refs serializadas — adicionar campo [SerializeField] numa tela SEM atualizar a factory correspondente e regerar a cena deixa a ref nula silenciosamente (todas as telas têm null-guards, o recurso simplesmente não aparece). (2) ORDEM DE INIT: UIManager.Init é chamado pelo GameBootstrap na ordem canônica (nunca auto-registro em Awake); CelebrationOverlay/ChestRevealOverlay usam Subscribe lazy com retry em OnEnable porque UnitManager/EconomySystem/RewardSystem podem não existir ainda — overlays novos devem copiar esse padrão. (3) BUS ESTÁTICO: GameEvents e os estáticos de cena (MainMenuController.s_pendingLevel, MetaBridge.s_seasonClaimedTier, Instances de overlay) sobrevivem a trocas de cena — sempre -= em OnDisable/OnDestroy; o cursor do passe NÃO persiste entre sessões (recompensas re-resgatáveis por sessão, decisão consciente documentada). (4) TELA PASSIVA: ResultScreen não pode ganhar lógica (cálculo de combo, resolução de próximo nível) — isso pertence ao GameUIController/sistemas; quebrar essa fronteira quebra os testes e o contrato documentado. (5) TIMESCALE: GameUIController é o dono do Time.timeScale (pausa) e ForceClosePause força 1f em fim de fase/troca de estado — um slow motion cinematográfico de morte de boss precisa coordenar com esse caminho ou será cancelado; transições de UI são unscaled e seguram timeScale 0. (6) PILHA DO UIMANAGER: Push esconde a tela do topo; CloseResult/ClosePause dependem de ReferenceEquals com CurrentScreen/CurrentOverlay — empilhar telas novas entre o resultado e o Pop pode deixar a pilha suja. (7) SAVEDATA: o dono é 'SISTEMAS' — MetaBridge deliberadamente DERIVA estado (passe, eventos) em vez de adicionar campos; uma BossCollection persistente exige negociar campo novo no SaveData ou usar derivação (boss derrotado ⇐ highestLevelCleared ≥ fase do boss). (8) DEFER DE 1 FRAME: a ResultScreen é mostrada 1 frame após OnLevelFinished para os commits rodarem antes — qualquer dado novo exibido nela deve estar pronto nesse mesmo contrato (listeners de dados antes da UI).

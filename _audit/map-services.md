# services
## SUMMARY
O subsistema de Services é uma camada de fachadas singleton MonoBehaviour (AdsManager, AnalyticsManager, IAPManager, RemoteConfigManager, AudioManager), todas IInitializable inicializadas pelo GameBootstrap na ordem do doc 12 §3.3. Os SDKs reais não existem: cada manager fala com uma interface declarada em Core (IAdsProvider, IAnalyticsProvider, IRemoteConfigProvider) implementada por providers Null (rewarded nunca pronto, analytics espelhado no console em DEV, remote config devolvendo defaults embutidos que espelham o CANON). A política de interstitial é uma função pura em Domain (InterstitialPolicy.ShouldShow: !adsRemoved && fase >= 6 && gap >= 3 fases && < 2 derrotas seguidas, com minLevel/levelGap vindos do Remote Config). O AnalyticsManager já define açúcar tipado para 34 nomes de evento (23 do BRIEF + 11 proprietários), mas só 6 estão de fato disparando em runtime (gate_selected, level_start, rewarded_ad_shown, rewarded_ad_completed, interstitial_shown, purchase_started) — o resto é API pronta aguardando fiação. O AudioManager é 100% dirigido por eventos do bus (GameEvents/JuiceEvents) com catálogo AudioCatalogSO de 14 slots + música por mundo, rate limits e pitch random. Estado persistente (adsRemoved, pacing, entitlements, sfxOn/musicOn) entra por BindSaveState com a instância viva do SaveData publicada no blackboard do GameBootstrap; UI e Gameplay nunca chamam Services direto — consomem hooks publicados no blackboard (RewardedAdReady, ShowRewardedAd, PurchaseProduct, SeasonPassOwned).

## EVENT FLOW
Boot: GameBootstrap inicializa providers Null no prefab [Services] e chama Init() de cada manager via IInitializable na ordem do doc 12 §3.3 (Save primeiro — publica SaveData vivo no blackboard). Regra de fronteira (doc 12 §2.3): Gameplay/UI/Meta NUNCA chamam Services; tudo flui pelo bus estático GameEvents/JuiceEvents (Core) ou por hooks no blackboard do GameBootstrap. Analytics: GameEvents.OnGateConsumed(GateResult{gate, rejected}) → AnalyticsManager.HandleGateConsumed → gate_selected; GameManager.LevelStarted(int) → level_start; AdsManager.ShowRewarded → rewarded_ad_shown e, no granted, rewarded_ad_completed; AdsManager.MaybeShowInterstitial → interstitial_shown; IAPManager.Purchase → purchase_started. Todos os demais 28 eventos têm método tipado mas ZERO call sites. Ads: GameEvents.OnLevelFinished(LevelResult{won, levelIndex}) → AdsManager.HandleLevelFinished → atualiza espelho de pacing (se sem bind) → MaybeShowInterstitial (lê RC, decide via Domain.InterstitialPolicy, zera pacing no save ANTES de exibir). UI consome rewarded via blackboard: GameBootstrap.Current.RewardedAdReady/ShowRewardedAd (preenchidos no AdsManager.Init). Audio: GameEvents.OnGateConsumed/OnSupplyOverflow(SupplyOverflow)/OnCrowdChanged(int,int)/OnMutationGained(MutationConfigSO)/OnBossPhaseChanged(BossPhase)/OnLevelFinished(LevelResult)/OnCurrencyChanged(CurrencyChange{type,amount}) + JuiceEvents.OnBossHitPulse(Vector3)/OnObstacleHit(Vector3) + GameManager.StateEntered(GameState) → handlers do AudioManager com rate limits e pitch random. IAP: UI lê GameBootstrap.Current.SeasonPassOwned()/PurchaseProduct(...); structs/payloads usados: GateResult, LevelResult, SupplyOverflow, CurrencyChange, BossPhase, GameState.

## RISKS
1) Fronteira de assemblies é regra dura (doc 12 §2.3): Gameplay/UI/Meta não referenciam Services — qualquer evento novo de boss/combo TEM de nascer como GameEvents/JuiceEvents em Core e ser assinado pelo manager; chamada direta quebra a direção de dependência. 2) Ordem de init §3.3: AnalyticsManager.Init assina GameManager.Instance.LevelStarted SÓ se GameManager.Instance já existir no momento do Init — novos managers/eventos têm a mesma armadilha de ordem; SaveSystem precisa rodar antes para o BindSaveState via blackboard funcionar (senão cai no modo espelho). 3) Bus estático sobrevive a troca de cena: todo handler novo precisa do padrão -= antes de += no Init e remoção no OnDestroy/Unsubscribe, senão duplica em re-Init de teste. 4) Pacing de interstitial tem DOIS modos (save ligado vs espelho) e o espelho replica manualmente as transições do SaveSystem.RecordLevelEnd — mudar o fluxo de derrota/revive (missão mexe nisso) exige manter os dois caminhos coerentes; o pacing zera ANTES de exibir (callback de SDK não é confiável) — não mover. 5) Convenções de taxonomia são contrato (doc 11 §3): snake_case, bool como int 0/1 via B(), globais session_id/level/world/app_version/seconds_in_session anexados em Send() sem sobrescrever valores explícitos — eventos novos que passem 'level'/'world' próprios devem saber disso. 6) NullAdsProvider.IsRewardedReady = false sempre: o botão DOBRAR da tela de vitória ficará oculto no MVP (nunca botão morto) — testes do fluxo de rewarded exigem fake injetado via Init(provider) (overloads existem exatamente para isso). 7) Assinaturas marcadas 'assinatura é contrato' (LogLevelStart(int), LogGateSelected(string,string), LogRewardedShown(string), LogInterstitialShown()) não podem mudar — adicionar overloads, não alterar. 8) NullRemoteConfigProvider.Defaults é a 'tabela EXATA do doc 12 §4.10' e RcKeys bane string mágica: nova chave sem const + default quebra a convenção e o 'jogável de fábrica'. 9) AudioManager tem rate limits afinados por handler (0.05–0.2s) e fonte pitched separada — sons de combate em rajada (hits de fraqueza, hordas) devem seguir o padrão ou viram metralhadora; catálogo é preenchido por tooling (MAR Tools/Build Audio) — slots novos precisam entrar nesse gerador também. 10) Consent gate: eventos pré-Init ficam em fila e 'denied' descarta — lógica duplicada em AnalyticsManager e NullAnalyticsProvider; mudanças devem manter os dois coerentes.
